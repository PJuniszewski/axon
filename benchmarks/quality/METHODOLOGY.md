# AXON Quality Benchmark — Methodology

## Why this exists

Phase 4 measured **token reduction** (74% native, 46% rule-based). That
proves AXON is cheaper. It does not prove AXON is **correct**: that the
compressed message carries the same meaning, that downstream agents
retrieve the right facts, that decoders don't hallucinate, that
CodecFit injection doesn't degrade reasoning.

This suite measures those orthogonal axes.

## Pre-registered tests

Every test is defined before any data is collected. Metrics are fixed.
We do not move the goalposts after seeing results.

| # | Test | Question | Primary metric | n |
|---|------|----------|----------------|---|
| 1 | `parser_fidelity` | Does AXON output always parse? | `% valid_parse` | 100 |
| 2 | `entity_preservation` | Do numbers/IDs/agents survive NL→AXON encoding? | `entity_recall` | 30 |
| 3 | `roundtrip_fidelity` | Does NL→AXON→NL preserve meaning? | LLM-judge similarity (0–10) | 20 |
| 4 | `hallucination_rate` | Does decoding invent facts? | facts_added per message | 20 |
| 5 | `context_retrieval` | Can a model answer Qs about prior AXON history? | accuracy vs NL baseline | 30 |
| 6 | `multihop_drift` | Does meaning drift across re-encoding chains? | similarity decay per hop | 10 |
| 7 | `reasoning_quality` | Does CodecFit in system prompt hurt CoT? | task accuracy Δ | 20 |
| 8 | `native_quality` | Does native AXON output preserve task fidelity? | task_success rate | 15 |
| 9 | `adversarial` | How does decoder handle malformed/ambiguous AXON? | refuse / hallucinate / error | 15 |

## Subject under test

For LLM-dependent tests, the subject is **Claude Sonnet** (the same model
class measured in `native_simulation.ts`). When run without an
`ANTHROPIC_API_KEY`, the harness can route LLM calls through Claude Code's
`Agent` tool — each subagent is an isolated, fresh-context instance of the
same model family. This is documented per-test in the run logs.

## What we deliberately do NOT measure (yet)

- **Cross-model consistency** — Haiku/Opus/GPT/Gemini decode rates. Single
  model env in this run.
- **Latency under load** — gateway routing throughput. This is an infra
  question, not a quality question.
- **Long-tail rare-symbol coverage** — only the 31 codebook symbols.
  Out-of-codebook symbol hallucination is captured in test 1.

## LLM-as-judge: the obvious caveat

Tests 3, 4, 5, 7 use Claude itself as a judge. This is biased by
construction — the model that writes AXON judges whether AXON preserves
meaning. We mitigate by:

1. **Separate context per role.** Encoder, decoder, and judge are
   different subagents with no shared history.
2. **Anchored rubrics.** The judge gets a 0–10 scale with explicit anchors
   (10 = "every fact, qualifier, and entity preserved"; 5 = "core facts
   preserved, qualifiers/quantifiers lost"; 0 = "fundamentally different
   meaning").
3. **Adversarial probes.** Test 9 deliberately includes cases where the
   correct answer is "refuse" — if the judge happily rates hallucinated
   expansions as 10/10, that's flagged.

A future external-model judge run (Gemini / GPT-4o) is the proper
robustness check. We mark this as known gap.

## Sample sizes

Small (n=15–30 per test). Power calculations for these metrics require
effect sizes we don't yet have priors on. This first run is **exploratory**:
the goal is to detect order-of-magnitude problems (e.g., 30%+ entity loss
on common messages), not to estimate effects to two decimal places.

Confidence intervals reported where applicable using Wilson score for
proportions, bootstrap for means.

## Reproducibility

Every test:

- Runs from a fixed seed corpus committed in `corpus/`
- Records the exact prompt sent to each subagent in `results/<test>/runs.jsonl`
- Reports raw per-sample data alongside aggregates
- Is re-runnable with `pnpm tsx benchmarks/quality/tests/<n>_*.ts`

## How to read the report

`results/REPORT.md` is generated by `aggregate.ts` after each test
completes. It includes:

- Headline number per test
- 95% CI where applicable
- Per-sample failure analysis (worst 3 cases verbatim)
- Cross-test correlations (e.g., does entity loss predict retrieval failure?)

If a test was skipped (e.g., no API key), the report says so explicitly
rather than substituting a fake number.
