## 1. Parser fidelity

**100.0%** of AXON outputs parsed cleanly (n=50).

- Rule-based encoder output: **100.0%** (30/30)
- Native LLM samples (synthetic, illustrative): **100.0%** (20/20)

> ⚠ The "native LLM samples" set is composed of plausible-looking outputs
> assembled by the benchmark authors, not collected from live API runs.
> Treat the 100% parse rate as a smoke test, not a measurement of how
> often real LLM emissions parse.

## 2. Entity preservation under rule-based encoding

Macro-averaged recall across **30** annotated messages:

| Category    | Recall |
|-------------|-------:|
| Numbers     | 88.4% |
| IDs         | 45.5% |
| Intent      | 40.0% |
| Qualifiers  | 24.9% |
| **Overall** | **48.5%** |

**Failure modes observed:**

- `qm-17` (existential-check, recall 0.0%):
  - NL:   `Check if any of the worker pods are in CrashLoopBackOff state.`
  - AXON: `! Check wkr pods Cra state`
  - missed IDs: worker, CrashLoopBackOff
  - missed intent: query (found `none` instead of expected symbol)
  - missed qualifiers: any

- `qm-22` (ambiguous-pronoun, recall 0.0%):
  - NL:   `Send the report to the orchestrator before it expires.`
  - AXON: `≡ report orch expires`
  - missed IDs: orchestrator
  - missed intent: request (found `none` instead of expected symbol)
  - missed qualifiers: before

- `qm-25` (metric-window, recall 22.2%):
  - NL:   `Error rate over the last 15 minutes is 4.2 percent, up from 0.3 percent over the prior hour.`
  - AXON: `⊗ ⟦Error rate last 15 min 42 pct 03 pct prior hour⟧`
  - missed numbers: 4.2, 0.3
  - missed intent: inform (found `none` instead of expected symbol)
  - missed qualifiers: over, up from

**Diagnosis.** The rule-based encoder has at least three identifiable
bugs that the headline 46% token-reduction number masks:

1. **Intent ordering bug** — the regex list classifies "Retry the failed
   payment" as ERROR (`⊗`) because `/failed/` is matched before the
   RETRY rule. Similarly "Deployment complete" lands on REQUEST (`!`)
   because `/deploy/` fires before the COMPLETE rule.
2. **Decimal-number loss** — `4.2 percent` becomes `42 pct` after
   punctuation stripping. The number is fundamentally wrong, not just compressed.
3. **Over-aggressive abbreviation** — `CrashLoopBackOff` becomes `Cra`;
   `orchestrator` is sometimes dropped entirely; `production` and
   `kubernetes` are kept verbatim in some cases but expected as
   `prod`/`k8s` here, dragging down the ID-recall macro.

## 3. Round-trip semantic fidelity

NL → encode(rule-based) → AXON → decode(LLM) → NL'
Judged by a separate LLM-as-judge on a 0–10 anchored rubric (strict).

- Mean score: **9.20/10** [95% CI bootstrap: 8.53–9.73]
- Median: **10/10**
- Distribution:
  - 9–10: **25** (83.3%)
  - 7–8:  2  (6.7%)
  - 4–6:  3  (10.0%)
  - 0–3:  0  (0.0%)

**Failure pattern.** Every case with score ≤6 lost a *qualifier* or
condition during round-trip:

- "only fails for EU users, only during peak hours" → "only" dropped twice
- "less than 30 days" → softened to "within 30 days"
- "timeout after 5 seconds if no response" → "if no response" clause dropped
- "all 47 tests" → "47 tests" (universal quantifier dropped)
- "Please" politeness markers consistently dropped

This is the same axis flagged by test 2: qualifiers are the weak link.
Round-trip preserves entities but degrades scope/condition.

## 4. Hallucination rate under decode

For the same 30 round-trip outputs as test 3, a separate judge
enumerated facts that appeared in the decoded NL but were NOT in the
original:

| Metric                       | Value |
|------------------------------|------:|
| Hallucinations per message   | 0.13 |
| Information loss per message | 0.43 |
| Modified facts per message   | 0.27 |
| Clean messages (0 added, 0 modified) | 19/30 (63.3%) |

**Important caveat.** All four "hallucinations" the judge flagged were
the decoder adding a label prefix like `Error:` or `Query:` to the
front of the natural-English expansion (the original NL did not contain
that label). These are not factual hallucinations — they are restatements
of the intent that was already encoded as the AXON performative symbol.

**Real factual hallucination rate: 0/30.**

The actual risk is on the other axis: 43.3% facts
lost per message — concentrated on the same qualifier-loss pattern that
tests 2 and 3 surface.

## 5. Context retrieval (needle-in-conversation)

5 multi-message agent threads × 6 probes × 2 forms (NL baseline vs AXON)
= 60 retrieval probes total.

| Form | Overall | Early | Mid | Late |
|------|--------:|------:|----:|-----:|
| NL   | 96.7% | 100.0% (n=11) | 88.9% (n=9) | 100.0% (n=10) |
| AXON | 96.7% | 100.0% (n=11) | 88.9% (n=9) | 100.0% (n=10) |
| Δ    | **0.0 pp** | | | |

**Interpretation.** Retrieval over a 8–10 message agent thread does not
degrade when the thread is AXON-encoded. The lone failure was identical
in both forms (a probe asking *why* something happened; the model gave a
defensible paraphrase that didn't exact-match the rubric — likely a
false negative in the scorer, not an information-loss event).

Caveat: probes are dominated by **unique identifiers** (PR numbers,
build IDs, dates, named services). These are the entities AXON
preserves best. A retrieval benchmark designed around qualifiers,
conditions, or hedged claims would likely show a different result — and
based on tests 2/3 that is exactly where AXON loses information.

| 1 | 10 | 6.10 | 7 | 3.90 |

**Interpretation.** Cumulative drop across 3 hops: -0.20 points.
Drift is small — most loss happens on the first compression; subsequent passes are nearly lossless once the message is already AXON-shaped.

## 7. Reasoning quality under CodecFit injection

20 reasoning problems given to two arms: baseline (no system prompt
prefix) vs. codecfit (148-token AXON primer in system prompt). Both told
to reason step-by-step and output a final numeric/short answer.

| Arm      | Overall | Easy | Med | Hard |
|----------|--------:|-----:|----:|-----:|
| Baseline | 85.0% | 100.0% | 87.5% | 50.0% |
| CodecFit | 85.0% | 100.0% | 87.5% | 50.0% |
| Δ        | **0.0 pp** | | | |

**Interpretation.** Injecting CodecFit into the system prompt did not
degrade chain-of-thought reasoning on this sample. Effect is
indistinguishable from noise at this n. This is the most reassuring
result in the suite — the worry that "agents reason worse when forced
to think in compressed symbols" is not supported here. A larger
benchmark on GSM8K-scale problems would tighten the bound.

## 8. Native AXON generation quality (A/B blind judge)

15 short agent-style tasks. Two arms generate responses:
- **baseline** — natural English
- **codecfit** — AXON (with CodecFit primer in system prompt)

A blind judge (does not see which arm produced which) scores both on
task fidelity and entity completeness on a 0–10 scale.

| Arm      | Task fidelity | Entity completeness | Total tokens |
|----------|--------------:|--------------------:|-------------:|
| Baseline | 10.00 | 10.00 | 380 |
| CodecFit | 8.20 | 8.93 | 273 |
| Δ        | -1.80 | -1.07 | -107 |

**Token reduction: 28.2%**.

**This contradicts the README's headline 74% native-mode figure.** The
74% was measured against deliberately verbose CrewAI/AutoGen-style
samples (the LLM was instructed to be verbose). Against an LLM that is
*already concise* in its NL baseline (because we asked it to be), the
saving collapses to 28.2%.

There is **also a quality cost**:
- Task fidelity drops by 1.80 points (out of 10)
- Entity completeness drops by 1.07 points

So the honest framing is: **native AXON saves tokens at the cost of
slight task degradation. The size of both depends entirely on how
verbose the NL baseline would have been.** If the baseline is already
terse, AXON is a worse-than-equal trade.

## 9. Adversarial / malformed AXON

15 broken or pathological inputs. Each has a designated correct
behaviour: `refuse`, `preserve_uncertainty`, or `best_effort`.

| Verdict | Count | Rate |
|---------|------:|-----:|
| Correct refuse / ok_hedge | 12 | 80.0% |
| Hallucinated meaning      | 1 | 6.7% |
| Over-refused              | 2 | 13.3% |

**The one hallucination** was on input `→@ ⟦rev PR#42⟧` — `@` with
no agent name. The decoder produced "Delegate to unspecified agent:
review pull request 42" with confidence 4. Borderline — the decoder
flagged low confidence and called the recipient "unspecified", which
is a hedge, but the input is fundamentally malformed (no routing
target) and the right answer is REFUSE.

**Conclusion.** The decoder is generally safe on broken input. It does
not confidently invent meaning. The two over-refuses (cases marked
`best_effort` where it refused anyway) are conservative, not harmful.

## Cross-test synthesis

Three findings replicate across independent tests:

1. **Entities (numbers, IDs) survive AXON well.** Test 2 shows 88%
   number recall, test 5 shows 97% retrieval accuracy on
   identifier-heavy probes. Native generation (test 8) preserves
   89.3% of expected entities.

2. **Qualifiers and scope are the weak link.** Test 2: 24.9% qualifier
   recall in the encoder. Test 3: every failure case ≤6/10 was a
   qualifier loss ("only", "all", "if no response", "less than" →
   "within"). Test 4: 0.43 facts lost per message, concentrated on the
   same axis.

3. **Token savings depend heavily on how verbose the NL baseline is.**
   - vs deliberately-verbose CrewAI-style baselines: 74% (from prior
     `native_simulation` benchmark)
   - vs concise NL baselines (test 8): 28.2%
   - The 46% rule-based encoder result sits in the middle and is the
     most defensible single number, but only if you assume agents would
     otherwise be verbose.

## What the README should change

Based on these results, the README's claims need three honest edits:

1. **Replace "74% native AXON output" with a conditional.** "74% on
   verbose agent outputs; ~28% when the NL baseline is already concise."
2. **Add a quality caveat.** Native AXON has measurable cost: ~1.8 pts
   task fidelity and ~1.1 pts entity completeness in this run.
3. **Document the qualifier-loss failure mode.** The protocol is
   lossy on scope and condition qualifiers. For agents whose contracts
   depend on "only", "all", "until", "approximately", AXON is not
   currently safe.

## Open questions for next round

  Claude-as-judge scores in tests 3, 4, 8?
- Do the encoder bugs (intent ordering, decimal stripping) account
  for most of the test 2 / test 3 degradation, or is the underlying
  protocol design also at fault?
- Does retrieval over qualifier-heavy probes (not just identifier
  probes) show the same 0 pp NL/AXON gap? Test 5 may have been too
  generous to AXON.
- What is the break-even token saving at which the 1.80 pt
  quality cost is acceptable? Today there is no formal answer.
